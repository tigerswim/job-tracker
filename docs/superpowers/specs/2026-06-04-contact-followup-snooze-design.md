# Contact Follow-up Snooze

**Date:** 2026-06-04  
**Status:** Approved

## Problem

The auto-followup system (`sync-google-interactions` Edge Function) creates a new reminder each day for any contact with an open interaction loop (unanswered email, meeting with no follow-up sent, gone quiet). Once a reminder fires (status = `sent`), the dedup check no longer blocks a new one — so reminders repeat daily until the contact replies or the interaction ages out of the horizon window. There is no way to tell the system "I don't need a reminder about this person right now."

## Solution

Add `followup_snoozed_until timestamptz` (nullable) to the `contacts` table. When set to a future timestamp, the sync function skips that contact when generating auto-followup reminders. When the timestamp passes, reminders resume automatically. "Indefinitely" is represented as `2099-12-31`.

Snooze can be set from two places:
1. The contact detail panel in the app (Network tab)
2. A snooze link embedded in the auto-followup reminder email

## Section 2: Backend — Followup Logic

**`runFollowups()`** in `supabase/functions/sync-google-interactions/index.ts`:

Before calling `detectOpenLoops`, fetch the set of currently-snoozed contact IDs:

```ts
const now = new Date()
const { data: snoozed } = await supa.from('contacts')
  .select('id')
  .eq('user_id', RUN_USER)
  .gt('followup_snoozed_until', now.toISOString())
const snoozedIds = new Set((snoozed ?? []).map(r => r.id))
```

Then filter loops after `detectOpenLoops`:

```ts
const loops = detectOpenLoops(rows ?? [], settings, now)
  .filter(l => !snoozedIds.has(l.contactId))
```

This keeps `detectOpenLoops` pure (no DB access, easier to unit test).

**Vendored copy:** `supabase/functions/sync-google-interactions/_shared/` — no changes needed to `followup-rules.ts` itself since the filter happens in the caller. The vendored drift check continues to pass.

## Section 3: Snooze from the Contact Panel

**Location:** `ContactModal` in `src/components/ContactList.tsx`

**UI:** A "Snooze follow-ups" button in the modal header (alongside the existing Edit button). Clicking opens an inline dropdown with:

- 1 week
- 1 month
- 3 months
- Indefinitely

Selecting an option calls `updateContact(id, { followup_snoozed_until: <iso> })` — no new API endpoint.

**Status indicator:** If `contact.followup_snoozed_until` is set and in the future, the modal shows a small badge ("Follow-ups snoozed until Jun 11") with a "Clear" link that sets `followup_snoozed_until` to null.

**`ContactModal` currently receives** a `Contact` object from parent state. Since `followup_snoozed_until` is now on the `Contact` interface and included in the select queries, it will be present. After a snooze update, the modal refreshes the contact in local state so the badge reflects the new value immediately.

## Section 4: Snooze from the Reminder Email

**New API route:** `src/app/api/contacts/[id]/snooze/route.ts`

`GET /api/contacts/[id]/snooze?duration=1w|1m|3m|indefinite&uid=<userId>&token=<hmac>`

- `uid` is `RUN_USER` (the hardcoded sync user ID), embedded in the URL when the link is generated by the Edge Function
- Validates HMAC-SHA256 of `contactId + ":" + duration + ":" + uid` signed with `SNOOZE_LINK_SECRET` env var
- Looks up the contact by `id` scoped to `uid` (RLS bypass via service role key, or just a `.eq('user_id', uid)` check)
- Sets `followup_snoozed_until` on the contact
- Returns a plain HTML confirmation page: "Got it — follow-ups for [Name] snoozed for [duration]."
- No auth session required (link is self-contained and HMAC-protected)

**Edge Function changes** (`sync-google-interactions/index.ts`):

- Read `SNOOZE_LINK_SECRET` from Deno env
- When inserting a new auto-followup reminder, generate four HMAC-signed snooze URLs and embed them in an HTML `email_body` instead of the current plain-text string
- `email_subject` stays as plain text (unchanged)

**Email format** (simple, functional):

```
[Reminder subject]

Snooze reminders for this contact:
· 1 week   · 1 month   · 3 months   · Indefinitely
```

**`SNOOZE_LINK_SECRET`:** 32-byte random hex. Added to:
- Netlify environment variables (for the API route)
- Supabase Edge Function secrets (for the Edge Function)
- `.env.local` (for local dev)

## Out of Scope

- Snooze from the reminder list UI (Reminders tab) — can be added later
- Per-tier snooze (e.g. snooze only "gone quiet" reminders) — YAGNI
- Email link expiry — the snooze timestamp on the contact handles staleness
- Multi-user support for the sync — pre-existing limitation, not introduced here
